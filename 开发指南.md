# 开发指南

本文档供 AI coding 工具进入本项目后快速建立上下文。优先按这里的项目约定执行，通用协作规则仍以会话或 `AGENTS.md` 中的全局规则为准。

## 项目地图

- `apps/control-plane`：Fastify 控制面，负责 agent 注册、路由、任务执行、调度、状态持久化和 API。
- `apps/web`：React + TanStack Router 前端，负责首页工作台、详情页、主题、布局偏好和 API 调用。
- `packages/shared-types`：前后端共享领域类型，包含 `AppState`、`DashboardData`、各业务状态和响应类型。
- `packages/agent-sdk`：agent manifest、执行请求、执行结果和 `defineAgent` / `defineAgentManifest`。
- `packages/agent-registry`：manifest 注册表，按 id、capability、trigger 查询 agent。
- `packages/agent-registry/src/sub-agents.ts`：子 Agent 集中配置，统一维护已注册 sub-agent 的 manifest 列表和首页模块定义。
- `packages/router-core`：主 agent 的启发式路由逻辑，按 manifest 的 trigger、tags、capabilities、name、description 选择候选。
- `packages/task-core`：任务记录和任务状态流转工具。
- `agents/*`：各子 Agent 实现，每个 agent 通常包含 `package.json`、`src/manifest.ts`、`src/index.ts` 和必要测试。
- `apps/control-plane/src/services/model-*`：模型供应商、模型配置、密钥引用和模型调用运行时。
- `apps/control-plane/src/services/browser-automation-*`：浏览器自动化 workflow、执行器和示例流程。
- `apps/control-plane/src/services/prompt-template-service.ts`：提示词模版分析、变量化和复用生成。
- `docs/superpowers/specs`、`docs/superpowers/plans`：已有功能设计和实施计划，适合查历史决策。

## 公共架构约定

- 工作区由根目录 `package.json` 的 npm `workspaces` 管理，范围是 `apps/*`、`packages/*`、`agents/*`。本项目默认使用 npm 和 `package-lock.json`，不要同时维护 pnpm workspace 配置。
- TypeScript 使用 strict 模式，路径别名集中在 `tsconfig.base.json`。新增跨包引用前，先确认已有别名或 package export。
- 控制面入口是 `apps/control-plane/src/app.ts`，运行入口是 `apps/control-plane/src/index.ts`。
- 前端路由入口是 `apps/web/src/router.tsx`，API 封装在 `apps/web/src/api.ts`。
- 本机小工具集中放在 `/tools`，每个工具使用独立子路由；磁盘操作由 `apps/control-plane/src/services` 下的独立 service 执行。
- 持久化状态默认写入 `.agent-zy-data/state.json`，状态结构以 `packages/shared-types/src/index.ts` 为准。
- 结构化事件日志独立写入 `.agent-zy-data/logs/events.jsonl`，不要塞进 `AppState` 或 `.agent-zy-data/state.json`。
- 子 Agent 不应直接改全局状态文件。执行结果通过 `AgentExecutionResult.domainUpdates` 返回，由 `apps/control-plane/src/services/store.ts` 统一落库。
- `packages/agent-registry/src/sub-agents.ts` 是当前 sub-agent 注册和首页模块映射的单一配置源。不要在多个文件各自维护一份 manifest 列表或首页模块默认值。
- manifest 的 `modulePath`、`manifestPath` 必须指向真实源码路径，不要虚构路径或依赖尚未存在的导出。
- 模型调用统一走 `apps/control-plane/src/services/model-runtime.ts`，不要在 agent、service 或前端里直接拼供应商请求。

## 首页模块尺寸约定

首页模块尺寸集中定义在 `apps/web/src/home-layout.ts`，任何新增首页模块都必须支持以下五档：

| size | 中文标签 | 网格尺寸 |
| --- | --- | --- |
| `max` | 最大 | `8x12` |
| `large` | 较大 | `4x6` |
| `medium` | 中等 | `8x3` |
| `smaller` | 较小 | `4x4` |
| `small` | 最小 | `4x2` |

相关规则：

- `HOME_MODULE_SIZE_OPTIONS` 定义可选值和中文标签。
- `HOME_MODULE_SIZE_GEOMETRY` 定义网格列数和行数。
- 收起态保持原宽度，行数统一变为 `1`。
- 预览尺寸由 `HOME_GRID_UNIT = 88` 和 `HOME_GRID_GAP = 14` 计算。
- 首页模块配置持久化 key 是 `agent-zy-home-layout-v1`，当前 payload version 是 `2`。
- 默认首页模块顺序和默认尺寸由 `HOME_MODULE_DEFINITIONS` 决定。
- 首页模块在工作台中必须交给 CSS Grid 自然铺排。不要在渲染层用 JS 计算并写入每个模块的内联 `gridColumn` / `gridRow` 坐标，否则模块会被固定到左侧堆叠，右侧大面积留空。尺寸类只应提供 `grid-column: span ...` 和 `grid-row: span ...` 这类跨度信息。

## 新增子 Agent 检查清单

新增一个子 Agent 时，按以下顺序检查：

1. 在 `agents/<id>-agent` 下创建 package，至少包含 `package.json`、`src/manifest.ts`、`src/index.ts`。
2. 在 `src/manifest.ts` 使用 `defineAgentManifest`，补齐 `id`、`name`、`description`、`version`、`capabilities`、`triggers`、`modulePath`、`manifestPath`、`tags`。
3. 在 `src/index.ts` 使用 `defineAgent`，实现 `execute(input: AgentExecutionRequest)`，返回 `AgentExecutionResult`。
4. 在 `packages/agent-registry/src/sub-agents.ts` 把新 agent 加入 `SUB_AGENT_ROUTES`。这里同时决定：
   `SUB_AGENT_MANIFESTS` 中有哪些已注册 sub-agent；
   `SUB_AGENT_HOME_MODULE_DEFINITIONS` 中有哪些首页模块及其默认顺序、默认尺寸、默认可见性。
5. 如新增领域状态或 API 响应，先更新 `packages/shared-types/src/index.ts`，再更新 store 的初始状态、normalize 逻辑和 dashboard 输出。
6. 如 agent 需要被主路由选择，确认 manifest 的 `triggers`、`tags`、`capabilities` 能被 `packages/router-core/src/index.ts` 命中。
7. 在 `apps/control-plane/src/app.ts`、或后续统一注册入口中接入 manifest，并按需在 `apps/control-plane/src/services/orchestrator.ts` 增加系统任务或 API 方法。
8. 如需要定时运行，在 `apps/control-plane/src/services/scheduler.ts` 增加调度常量和触发逻辑，并补调度测试。
9. 如需要前端入口，在 `apps/web/src/api.ts`、`apps/web/src/router.tsx`、相关页面或首页模块中接入。
10. 补 agent 自身测试。若涉及集中配置，同时补 `packages/agent-registry/src/sub-agents.test.ts`；若涉及控制面、路由或前端布局，同时补对应测试。

已有 agent 可作为参照：

- `ledger-agent`：用户触发、写 ledger domain update。
- `news-agent`：系统任务、信源管理、刷新、分析和 API。
- `topic-agent`：定时生成、dashboard 展示、独立页面。
- `history-agent`：定时推送通知，返回 persistent notification payload。
- `cinematic-agent`：用户触发，生成电影感分镜、连续性线索和人物、道具、场景参考提示词。
- `classic-shot-agent`：用户触发，拆解有明确出处的经典电影镜头，或基于上传视频抽帧生成 AI 视频复刻分镜提示词。
- `summary-agent`：个人日、周、月、年复盘，前端通过 `/summaries` 页面和 API 做草稿、归档、导入导出。
- `schedule-agent`：今日待办和夜间回顾，由调度器在本地 22 点触发 nightly review。
- `browser-automation-agent`：浏览器自动化能力描述；实际 workflow 的增删改查和执行由控制面 browser automation service 负责。

## 新增首页模块检查清单

新增首页模块时，按以下顺序检查：

1. 在 `apps/web/src/home-layout.ts` 更新 `BuiltInHomeModuleId` 和 `HOME_MODULE_DEFINITIONS`。
2. 如模块对应一个 sub-agent，优先同步检查 `packages/agent-registry/src/sub-agents.ts` 中的 `homeModule` 配置是否一致，避免首页模块定义和 sub-agent 配置分叉。
3. 为新模块选择默认 `defaultSize`，必须是 `max`、`large`、`medium`、`smaller`、`small` 之一。
4. 在 `apps/web/src/components/dashboard-page.tsx` 的 `getModuleSummary` 增加折叠态摘要。
5. 在 `renderHomeModuleContent` 增加模块内容渲染。
6. 确认内容在五档尺寸和收起态下都能合理显示。紧凑尺寸不要依赖大屏布局。
7. 如模块需要独立页面，更新 `apps/web/src/router.tsx` 和顶部导航 `railItems`。
8. 如模块依赖新数据，优先让 `DashboardData` 返回必要摘要，避免首页直接拼多个请求。
9. 修改首页布局或模块尺寸样式时，确认没有引入模块级内联坐标排布；保持 CSS Grid 根据模块跨度自然换行和铺满可用宽度。
10. 更新 `apps/web/src/home-layout.test.ts`，覆盖默认顺序、默认尺寸、尺寸几何或迁移逻辑变化。

## 状态、API 和通知约定

- 共享状态先定义类型，再在 `createInitialState` 中补默认值。
- 读 dashboard 走 `GET /api/dashboard`，实时更新走 `/api/stream` 的 SSE。
- 控制面 API 统一在 `apps/control-plane/src/app.ts` 注册，业务执行优先落到 orchestrator。
- 前端 fetch 包装统一放在 `apps/web/src/api.ts`，失败时抛出明确错误。
- 通知类型定义在 `NotificationKind`，agent 只返回通知内容，控制面负责生成 id、时间和 taskId。
- 长期展示通知使用 `persistent: true`，取消通知走 `DELETE /api/notifications/:id`。
- 首页布局 API 是 `GET /api/home-layout` 和 `PUT /api/home-layout`，浏览器本地布局 key 是 `agent-zy-home-layout-v1`。
- 模块独立页面路由集中在 `apps/web/src/router.tsx`，新增页面时同步检查顶部导航、首页模块和对应 API 封装。

## 模型配置和模型调用约定

- 模型供应商定义在 `apps/control-plane/src/services/model-providers.ts`，当前支持 `modelscope`、`deepseek`、`openai`、`doubao`、`ollama` 和 `openai-compatible`。
- 模型配置状态保存在 `AppState.modelSettings`，共享类型是 `ModelProfile`、`ModelProviderDefinition`、`ModelSettingsState` 和 `ModelSettingsDashboard`。
- 模型配置 API 包括：
  `GET /api/model-providers`；
  `GET /api/model-profiles`；
  `POST /api/model-profiles`；
  `PATCH /api/model-profiles/:id`；
  `DELETE /api/model-profiles/:id`；
  `POST /api/model-profiles/:id/test`；
  `POST /api/model-profiles/default`；
  `POST /api/model-profiles/purpose-default`；
  `POST /api/model-profiles/agent-default`。
- API Key 不写入 `state.json`。环境变量密钥使用 `env:*` 引用，本地保存的密钥由 `createModelSecretsRepository` 管理，前端只拿到 `hasApiKey`、`maskedKey` 和 `apiKeySource` 这类状态。
- 默认模型选择顺序是：显式 `profileId`、agent 默认配置、purpose 默认配置、全局默认配置。找不到启用的 profile 时，模型运行时抛明确错误。
- 模型调用统一使用 `ModelRuntime` 的 `chat`、`generateText`、`embedding` 或 `execute`。需要 JSON 输出时传 `responseFormat: "json"`；需要图片输入时 profile 必须包含 `vision` capability。
- 默认 ModelScope 配置可由 `.env` 中 `MODELSCOPE_API_KEY`、`MODELSCOPE_BASE_URL`、`MODELSCOPE_MODEL` 初始化。其他供应商环境变量以 `.env.example` 为准，不要把真实密钥写进文档或源码。

## 结构化日志约定

- 控制面日志服务位于 `apps/control-plane/src/services/event-log-service.ts`，按 JSONL 追加写入 `.agent-zy-data/logs/events.jsonl`。
- 日志 API 是 `GET /api/logs`、`POST /api/logs/client-events` 和 `DELETE /api/logs`。`/api/logs`、`/api/stream`、健康检查和系统状态检查不参与 API hook 记录，避免递归和噪声。
- 日志用于排障，不是业务状态。写日志失败不得阻断 API、任务、Agent 或模型调用。
- 默认保留 14 天。追加日志时清理过期记录；查询遇到损坏 JSONL 行时跳过并返回警告。
- `apiKey`、`authorization`、`token`、`secret`、密码和凭证类字段必须脱敏。模型调用只保存供应商、模型、耗时、状态、字符数和截断摘要，不保存完整输入输出。
- 前端只上报生成、刷新、同步、重启和模型测试等关键用户操作。普通导航、复制、展开等低价值交互不要写入日志。

## 本机工具约定

- 工具首页路由是 `/tools`。照片和视频名称修改工具沿用兼容路由 `/tools/photo-renamer`，对应 API 是 `POST /api/tools/photo-renamer/preview`、`POST /api/tools/photo-renamer/execute` 和 `POST /api/tools/photo-renamer/undo`。
- 工具目录当前包含：
  `/tools/photo-renamer`；
  `/tools/browser-automation`；
  `/tools/prompt-templates`；
  `/tools/file-organizer`。
- 本机磁盘工具必须先预览再执行。执行和撤销使用短时一次性令牌，并在真正修改文件前重新校验文件状态和目标路径占用情况。
- 磁盘批量重命名必须使用临时名称中转；中途失败时尽力回滚本批次，禁止直接覆盖已存在文件。
- 本机磁盘工具只接受本机浏览器来源。结构化日志可以记录动作、数量和耗时，不得记录用户目录完整路径或操作令牌。
- 照片和视频名称修改工具递归扫描目录但不跟随符号链接。照片支持 `jpg`、`jpeg`、`png`、`webp`、`heic`、`heif`，时间优先读取 EXIF。视频支持 `mp4`、`mov`、`m4v`、`avi`、`mkv`、`webm`，时间优先通过本机 `ffprobe` 读取容器中的 `creation_time`。元数据缺失或 `ffprobe` 不可用时回退到文件修改时间。
- 预览接口支持可选的 `mediaScope`：`images` 只处理图片，`videos` 只处理视频，`all` 同时处理图片和视频。未传递时默认使用 `all`，保持旧调用兼容。
- 文件整理工具对应 API 是 `POST /api/tools/file-organizer/preview`、`POST /api/tools/file-organizer/execute` 和 `POST /api/tools/file-organizer/undo`。递归扫描普通文件但不跟随符号链接，支持按时间或按类型移动文件并创建目标文件夹；执行和撤销都不得删除文件或清理创建出来的目录。
- 提示词模版工具对应 API 是 `GET /api/tools/prompt-templates`、`POST /api/tools/prompt-templates`、`PATCH /api/tools/prompt-templates/:id`、`DELETE /api/tools/prompt-templates/:id` 和 `POST /api/tools/prompt-templates/:id/apply`。创建时会尝试用模型提炼 `{{变量key}}`，模型失败时仍保存原提示词并把 `analysisStatus` 标为 `failed`。
- 浏览器自动化状态在 `AppState.browserAutomation`，包含 `workflows`、`runs`、`triggerRules` 和 `lastUpdatedAt`。运行记录最多保留最近 50 条。
- 浏览器自动化 workflow 步骤类型定义在 `packages/shared-types/src/index.ts`，包括 `openUrl`、`click`、`type`、`press`、`delay`、`extract`、`waitForCondition` 和 `ifElse`。视觉定位和页面条件判断由执行器结合浏览器观测与模型运行时处理。
- 浏览器自动化 API 是 `GET /api/browser-automation`、`POST /api/browser-automation/workflows`、`PATCH /api/browser-automation/workflows/:id`、`DELETE /api/browser-automation/workflows/:id`、`POST /api/browser-automation/workflows/:id/run`、`POST /api/browser-automation/runs/:id/stop` 和 `POST /api/browser-automation/trigger-rules`。
- 浏览器自动化触发规则通过 `task.completed` 事件匹配任务，可按 `agentId`、`status`、`trigger` 和 `summaryIncludes` 触发 workflow。
- 外部链接打开统一走 `POST /api/open-url`，只接受 `http` 或 `https` URL，并通过 `browser-opener` service 执行。

## 模块数据同步

- 历史知识、梦幻西游和浏览器自动化页面提供显式“同步数据”按钮。日常编辑仍实时写入本机 `.agent-zy-data`，只有点击按钮才会访问 Git 远端。
- 启用前必须先把远端仓库设为 Private，并设置 `AGENT_ZY_DATA_SYNC_ENABLED=true`。按钮使用本机现有 Git 用户信息和 SSH/credential，不保存或回显凭据。
- 同步数据固定提交到远端 `agent-zy-data` 分支的 `sync-data/<module>/`。普通代码分支的 `git pull` 不会导入这些数据；每台机器需要在对应模块点击按钮执行双向同步。
- 首次上线应在数据最完整的机器依次同步历史、梦幻和浏览器模块，再到其他机器点击同步。首次无基线时，本地缺失不视为删除。
- 历史模块同步生成内容、选题归档和稳定的小红书指标；梦幻模块同步所有账本、库存、资产及游戏币批次；浏览器模块只同步 workflow 和触发规则。模型密钥、Chrome profile、浏览器运行日志、截图和提取结果永不进入快照。
- 同一记录双端修改会返回冲突弹窗，必须逐条选择本地或远端。禁止 force push；非快进 push 会重新 fetch 并完整计算一次。
- 同步 API 是 `GET /api/data-sync/status` 和 `POST /api/data-sync/:module`，其中 module 只允许 `history`、`mhxy`、`browser-automation`。

## 复盘和视频生成约定

- 总结模块状态在 `AppState.summary`，类型包括 `SummaryEntry`、`SummaryState` 和 `SummaryDashboard`。支持 `daily`、`weekly`、`monthly`、`yearly` 四类复盘。
- 总结 API 是 `GET /api/summaries`、`GET /api/summaries/:id`、`POST /api/summaries`、`PATCH /api/summaries/:id`、`DELETE /api/summaries/:id`、`POST /api/summaries/generate-draft`、`POST /api/summaries/export` 和 `POST /api/summaries/import`。
- 经典镜头视频上传 API 是 `POST /api/classic-shots/generate-from-video`。仅支持 `mp4`、`mov`、`webm`，最大 100MB，`frameCount` 允许 3 到 8；上传文件只写入 `.agent-zy-data/tmp/classic-shots/<taskId>` 临时目录，生成结束后清理。
- 电影感分镜模块 API 是 `GET /api/cinematic`、`POST /api/cinematic/projects`、`PATCH /api/cinematic/projects/:id`、`DELETE /api/cinematic/projects/:id` 和 `POST /api/cinematic/generate`。分镜项目可包含 `scenePlan`、`continuity` 和 `referenceAssets`。
- 调度器在 `apps/control-plane/src/services/scheduler.ts`。当前包括：启动和每 30 分钟刷新 news；本地 22 点触发夜间回顾；每周一 08:00 生成上周账本周报；每月 1 日 08:05 生成上月账本月报。

## 开发规范

- 保持小而可 review 的 diff，不做与任务无关的重构。
- 不提交 secrets、token、私钥、`.env` 实际值或凭证。需要密钥时使用环境变量。
- 不新增无请求的 analytics、telemetry 或外部网络调用。
- 不发明 API、配置或文件路径。无法确认时先用 `rg` 搜索真实实现。
- 新增行为要补测试。纯文档变更不需要跑完整测试，但要校验引用路径和值。
- 注释只解释不明显的意图，不重复代码表面含义。
- 前端新增 UI 应沿用现有组件、主题和布局方式，避免把首页模块写成只适配单一尺寸的实现。

## 前端主题和色值约定

管理页曾出现过一次日间模式色值污染问题：新增管理页卡片和背景图预览时，样式中直接写入了夜间模式使用的深色 `rgba(14, 31, 50, ...)`、`rgba(8, 19, 35, ...)`、`rgba(7, 17, 29, ...)` 等背景色，同时全局背景还叠加了深色遮罩。结果是 `body[data-theme="day"]` 的主题变量没有被修改，但页面视觉仍然被新增深色样式覆盖，日间模式看起来发灰、发暗。

原因是新增 UI 没有完全走主题变量，而是把夜间模式的局部视觉当成通用样式写在了普通选择器里。普通选择器会同时作用于日间和夜间，除非再写日间覆盖规则；这种做法容易遗漏，也会让后续维护者误以为主题变量没有问题。

解决方式：

- 管理页使用独立的 `--manage-*` 变量描述页面级色值，包括 `--manage-surface`、`--manage-surface-strong`、`--manage-surface-hover`、`--manage-control`、`--manage-control-soft`、`--manage-line`、`--manage-line-strong`、`--manage-text`、`--manage-muted`、`--manage-shadow`。
- `:root` 定义夜间模式管理页色值，`body[data-theme="day"]` 定义日间模式管理页色值。组件样式只引用这些变量，不直接写夜间模式背景色。
- 自定义背景图只通过 `--custom-scene-backdrop` 叠加到 `--scene-backdrop` 前面，不再添加全局深色遮罩。需要提升图片文字可读性时，只在图片预览或图片已启用的局部元素上加遮罩。
- 新增或调整前端样式时，禁止把某个主题下的硬编码背景色直接放到跨主题选择器里。确实需要固定颜色时，必须说明它不受主题影响，例如图片上白字遮罩。
- 修改主题相关样式后，至少检查一次 `body[data-theme="day"]` 和默认夜间模式下的同一区域，确认背景、文字、边框和 hover 状态都来自对应主题变量。

## 常用命令

如果 shell 里的 `npm` 被 Codex app 自带 Node 接管，可能触发 Rollup 原生可选依赖签名问题。优先用本机 nvm Node 路径运行：

```bash
PATH=/Users/tal/.nvm/versions/node/v24.15.0/bin:$PATH npm test
PATH=/Users/tal/.nvm/versions/node/v24.15.0/bin:$PATH npm run typecheck
PATH=/Users/tal/.nvm/versions/node/v24.15.0/bin:$PATH npm run build:web
```

```bash
npm test
npm run typecheck
npm run build:web
npm run build
npm run dev:control-plane
npm run dev:web
```

按影响范围优先运行最快检查：

- 改 agent：`npm test -- agents/<id>-agent/src/index.test.ts`
- 改控制面 API 或调度：`npm test -- apps/control-plane/src/app.test.ts apps/control-plane/src/services/scheduler.test.ts`
- 改首页模块尺寸或布局：`npm test -- apps/web/src/home-layout.test.ts`
- 改共享类型或跨包接口：`npm run typecheck`
- 改前端页面或样式：`npm run build:web`
