# 工作日自动提交代码分支级审计设计

## 目标

现有自动化只检查当前分支，因此会漏掉本地其他分支上的未提交改动或待推送提交。

新的自动化目标不是“提交当前 checkout”，而是“审计整个本地仓库的分支集合”，并在单次运行内依次处理所有可安全处理的本地分支：

- 发现未提交修改时自动提交。
- 发现待推送提交时自动推送。
- 发现只有本地、尚未建立远端跟踪的分支时，自动首次推送并建立 upstream。
- 发现需要人工判断的风险时，仅阻塞对应分支，并继续处理其他安全分支。

## 第一性原理

这个任务的真实对象不是工作区，而是“本地分支及其与远端的同步状态”。

对任意本地分支，自动化只需要回答 4 个问题：

- 这个分支当前能否被安全检出并检查。
- 这个分支是否存在未提交修改。
- 这个分支是否存在待推送提交。
- 这个分支是否包含必须由人判断的风险。

因此流程必须先枚举分支，再对每个分支执行统一状态机，而不是从当前分支外推整个仓库状态。

## 范围

本次设计只调整“工作日自动提交代码”自动化的执行策略与输出格式。

包含：

- 扫描所有本地分支。
- 在同一次运行中逐分支处理提交与推送。
- 支持首次推送本地新分支到 `origin/<branch>` 并建立 upstream。
- 汇总每个分支的处理结果。

不包含：

- 自动拉取、自动 rebase 或自动解决远端冲突。
- 改写历史，如 `reset`、`rebase --interactive`、`push --force`。
- 跨仓库远端选择策略；首次推送仍只使用 `origin`。

## 分支审计流程

### 1. 初始化

- 记录起始分支名。
- 校验当前仓库不是 detached HEAD。
- 校验至少存在一个远端；首次推送依赖 `origin`。
- 枚举所有本地分支。

若初始化阶段已经处于 detached HEAD，整次运行停止并报告，因为无法定义稳定的“恢复到起始分支”动作。

### 2. 逐分支处理

对每个本地分支依次执行：

1. 判断当前工作区是否允许安全切换到目标分支。
2. 若不能安全切换，则将该分支标记为 `blocked`，记录原因，继续下一个分支。
3. 若可切换，则检出目标分支。
4. 读取该分支状态：
   - 工作区是否有未提交修改。
   - 是否存在冲突标记。
   - 是否存在疑似敏感文件或敏感内容。
   - 是否存在 upstream。
   - 若存在 upstream，当前分支相对 upstream 的 ahead / behind 状态。
5. 根据状态执行提交、推送或跳过。

### 3. 收尾

- 恢复到起始分支。
- 输出按分支聚合的结果：
  - `committed`
  - `pushed`
  - `skipped`
  - `blocked`
- 对已处理分支输出提交哈希、提交信息和推送目标。

## 安全切换规则

分支级批处理的核心风险不在提交，而在跨分支切换。

因此自动化在尝试检出目标分支前，必须判断：

- 当前工作区是否已经干净。
- 若不干净，这些未提交修改是否属于当前已检出分支，并且是否会被先处理完成。
- 检出目标分支是否会因未提交修改被 Git 拒绝。

设计约束：

- 自动化只在当前分支已经被处理到“可安全切换”后，才进入下一个分支。
- 若某个分支处理后仍留下未提交修改，则该分支记为 `blocked`，整次流程不再继续切换，以免污染后续分支判断。
- 不允许使用 stash 作为默认切换手段，因为它会引入额外隐式状态和恢复复杂度。

## 分支状态机

每个分支只落入以下 4 类结果之一：

### `skipped`

满足以下任一条件：

- 工作区干净，且不存在待推送提交。
- 分支没有任何可执行动作。

### `committed`

满足以下条件：

- 存在未提交修改。
- 不包含敏感内容、冲突标记或其他人工阻塞因素。
- 成功生成提交并创建 commit。

`committed` 只表示已创建新提交；若随后还发生推送，则最终结果可同时记录为“已提交并已推送”。

### `pushed`

满足以下任一条件：

- 工作区干净，但存在待推送提交，且推送成功。
- 本次运行刚创建提交，随后推送成功。
- 本地新分支没有 upstream，但成功推送到 `origin/<branch>` 并建立 upstream。

### `blocked`

满足以下任一条件：

- 检出目标分支前，Git 状态不允许安全切换。
- 发现 `.env`、凭据、令牌、私钥或其他疑似敏感文件。
- 发现疑似敏感内容。
- 发现合并冲突或冲突标记。
- 当前分支不存在 upstream，且远端 `origin` 缺失。
- 推送前发现分支 behind upstream，需要人工处理同步策略。
- 提交或推送过程中出现无法自动判断的 Git 异常。

## 敏感内容与风险判断

自动化继续沿用现有安全边界，但范围从当前分支扩展到所有待处理分支。

必须阻塞的对象包括：

- 文件名或路径明显属于 `.env`、密钥、证书、私钥、token、credential。
- 代码或文本中出现疑似密钥、访问令牌、私密配置。
- 冲突标记，如 `<<<<<<<`、`=======`、`>>>>>>>`。

设计要求是“宁可误报阻塞，也不要误提敏感内容”。

## 推送策略

### 已存在 upstream

- 若 ahead 为 0，则无需推送。
- 若 ahead 大于 0 且 behind 为 0，则直接推送到既有 upstream。
- 若 behind 大于 0，则标记为 `blocked`，因为自动化不负责决定 merge、rebase 或 force push。

### 不存在 upstream

- 若没有任何需要推送的提交，则允许 `skipped`。
- 若存在待推送提交，则必须先确认存在 `origin`。
- 若存在 `origin`，执行首次推送到 `origin/<branch>` 并设置 upstream。
- 若不存在 `origin`，标记为 `blocked`。

## 提交策略

- 只在该分支存在未提交修改且通过安全检查时创建提交。
- 提交前先基于实际 diff 范围生成简洁明确的提交信息。
- 不提交敏感文件。
- 不混入其他分支的工作区残留。

提交信息不要求跨分支统一风格，但要求能准确反映该分支本次变更主题。

## 输出格式

最终输出从“当前分支状态”升级为“仓库分支审计结果”。

输出应包含：

- 起始分支。
- 已扫描本地分支总数。
- 每个分支的结果类别。
- 对已提交分支：提交哈希与提交信息。
- 对已推送分支：推送目标。
- 对阻塞分支：阻塞原因。

如果所有分支都既无未提交修改也无待推送提交，则直接结束，不产生“无事可做”通知。

## 错误处理与恢复

- 任一分支失败后，自动化继续尝试处理后续可安全处理的分支。
- 若失败导致当前工作区无法继续安全切换，则停止后续分支遍历，并在最终报告中说明“后续分支未继续处理”的原因。
- 结束前尽最大努力恢复到起始分支；若恢复失败，必须在结果中明确报告。

## 验证与验收

需要覆盖的验证场景：

- 当前分支干净，但其他本地分支存在待推送提交。
- 当前分支干净，但其他本地分支存在未提交修改。
- 某个本地分支无 upstream，但存在本地提交，能够首次推送并建立 upstream。
- 某个分支 behind upstream，被正确阻塞且不自动推送。
- 某个分支包含冲突标记，被正确阻塞。
- 某个分支包含疑似敏感文件，被正确阻塞。
- 某个分支阻塞后，其余安全分支仍继续处理。
- 全部分支都无修改且无待推送提交时，任务静默结束。
- 处理完毕后恢复到起始分支。

验收标准：

- 自动化不再漏掉“非当前分支”的未提交改动或待推送提交。
- 自动化不会因为一个阻塞分支而放弃所有其他安全分支。
- 自动化不会对 behind upstream、冲突或敏感内容做未经授权的自动决策。
