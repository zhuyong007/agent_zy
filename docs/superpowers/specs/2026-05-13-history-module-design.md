# 历史知识模块去重与首页卡片设计

## 目标

- 历史知识生成成功后，将知识点写入本地文件存档，避免后续再次生成完全相同的知识点。
- 只重设计首页里的“历史知识”模块卡片，不修改 `/history` 详情页、不修改管理页预览、不修改全局样式体系。
- 首页卡片需要适配现有 `max`、`large`、`medium`、`smaller`、`small` 五种尺寸，并且每种尺寸展示适合自身面积的信息密度。

## 生成去重

- 在 `agents/history-agent` 增加历史知识档案文件，例如 `data/history/topic-archive.json`，用于持久化已生成过的知识点。
- 档案至少记录：`topic`、首次生成时间、最近生成时间、生成次数。
- 生成前先读取档案，优先选择未出现过的主题；如果内置主题库全部出现过，再按“最久未生成优先”回退，而不是固定按日期重复命中同一主题。
- 生成成功后再写回档案，确保只有真正生成完成的知识点才进入去重记录。
- 去重只基于知识点主题文本，不引入额外网络依赖，也不改变通知和 dashboard 的现有数据结构。

## 后端实现

- 在 `agents/history-agent/src/index.ts` 抽出主题归档读写和去重选题逻辑，保持现有 `generateWithModelScope` 负责内容生成，避免把文件 I/O 和模型请求耦合在一起。
- 新增纯函数处理以下职责：
  - 读取档案文件，不存在时返回空档案。
  - 根据内置主题列表和档案记录选择本次主题。
  - 生成成功后合并档案并写回文件。
- 保持 `HistoryPostPayload`、通知结构和 `historyPush.lastTriggeredDate` 不变，避免影响控制面和前端读取。
- 对手动生成和定时生成使用同一套去重策略，保证行为一致。

## 首页卡片视觉方向

- 采用 A 方案“策展档案卡”。
- 气质像历史馆展签与夜间档案盒：深墨色基底、克制的铜金色点缀、强标题、弱装饰、强调编目感和时间感。
- 不改全局 token，不新增会影响其他模块的基础类；样式只挂在首页历史卡片已有类名及其尺寸修饰类上。
- 动效保持克制：整体轻微抬升、编目刻度线高亮、拆卡层次轻微错位，不引入会影响全局性能的复杂动画。

## 五种尺寸的内容策略

### `max`

- 展示：标题、更新时间、主动生成按钮、摘要、三项统计、最多 5 条拆卡、正文预览。
- 角色：完整主模块，承担内容解释和点击引导。

### `large`

- 展示：标题、更新时间、主动生成按钮、摘要、三项统计、最多 4 条拆卡、较短正文预览。
- 角色：信息接近完整，但版式更紧凑。

### `medium`

- 展示：标题、更新时间、摘要、最多 3 条拆卡、单段正文摘录。
- 角色：优先保留阅读欲和主题辨识度，减少指标堆叠。

### `smaller`

- 展示：标题、更新时间或存档数、最多 2 条拆卡标题。
- 角色：像“今日选题卡”，一眼可读。

### `small`

- 展示：标题、1 条拆卡、极简状态信息。
- 角色：在最小面积下仍能明确这是“历史知识”而不是通用消息卡。

## 前端实现

- 继续沿用 `getHistoryHomePreviewRule` 控制五档尺寸的可见内容，但将规则补充到更明确的视觉分层需求。
- 首页历史卡片组件保留现有数据读取和交互逻辑，只调整结构层次和局部 class，以支持新的视觉样式。
- 将首页历史卡片样式收敛到模块专属区域，重点改造：
  - 头部信息层级
  - 摘要区和统计区的视觉关系
  - 拆卡列表的层次和密度
  - 不同尺寸下的留白、字号和隐藏策略
- `/history` 详情页相关的 `.history-stage`、`.history-archive` 等样式不作为本次改动目标，避免扩大范围。

## 错误处理

- 档案文件读取失败时，记录错误并回退为空档案，不阻塞首次生成。
- 档案文件写入失败时，本次任务应返回失败，避免页面显示“已生成”但归档未落盘。
- 若所有主题都已生成过，允许按“最久未生成”策略重复，但仍要更新档案中的最近生成时间和次数。

## 验证

- Agent 测试覆盖：
  - 首次生成时创建档案。
  - 已有档案时跳过重复主题。
  - 所有主题都已使用后按最久未生成回退。
  - 写档案失败时任务失败。
- 前端测试覆盖：
  - 五种尺寸下的历史卡片预览规则仍返回预期内容数量。
  - 首页历史卡片在无数据和有数据时都能正常渲染。
- 运行最快的相关检查，至少覆盖 `history-agent` 测试和 `apps/web` 相关测试。
